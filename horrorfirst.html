The Game

A boy falls asleep holding three of his teddy animals — and wakes up inside his own dream to find that three bosses have stolen them. He has to fight his way through the dream world, beat each boss, and get his teddies back.

What I Built

  • Save/persistence system — saves and restores full game state: player health, collected teddy animals, and boss-defeat flags (defeated bosses don't respawn on reload)
  • Boss fight system — main boss + mini-boss encounters, AI built manually without Unreal's State Tree/Behavior Tree tools
  • Inventory system
  • Teddy bear collection logic — pickup and quest-tracking for the three stolen animals
  • Projectile system — the character's finger-shooting attack
  • Object physics — shootable objects react and launch on impact
  • Character movement
  • Animation integration
  • Gameplay UI integration + UI design
  • Some scene lighting

Technical Challenge: Full Game State Persistence

Most student projects save one or two things — usually just player stats. This system had to track three different kinds of state at once: player health, which of the three teddy animals had been collected, and which bosses were already defeated — all surviving a save/load cycle correctly together, so the world came back exactly as the player left it.

Every destroyable actor in the game carries an isDestroyed tag. Once an actor is destroyed, its name gets added to a string array tracking destroyed actors. A SaveData Blueprint holds all the variables that need to persist, and a BP_QuestManager reads the current state of the game, builds a save game object from it, and passes that into the save game Blueprint. On load, the same path runs in reverse — the save game object is read and all the data is restored. In total, the system saves quest data, health, stamina, destroyed actors, and aim sensitivity.

The real challenge wasn't the save/load mechanics themselves so much as making sure the world came back correctly — this was new territory for me at the time. The game is linear: you fight bosses one door at a time, and a progression obstacle blocks the next door until the current boss is defeated. That meant the save system had to correctly restore not just "is this boss dead," but the downstream state that depended on it — the cutscene having played, the obstacle being cleared, the next door being accessible — all matching what the player had actually completed, not just resetting to a default state on reload.

Technical Challenge: Boss AI Without State Trees

At the time, I didn't know about Unreal's State Tree or Behavior Tree systems, so the boss and mini-boss logic was built by hand. The boss — a "BedMonster" — has two large arms extending from either side of the bed and can't move from that spot, so the AI couldn't rely on chase or pathing logic. Instead, I built it around two layers of collision:

Hand-shape detection

Each hand has a set of individual collision boxes mapped to its actual shape (fingers, palm) rather than one bounding box covering the whole arm. An animation Blueprint drives the arm through its attack, and if any of those hand-shaped boxes connect with the player, it deals damage with a short cooldown window before it can register another hit.

Zone-based triggering

The player's combat area is split down the middle into two collision zones, one per arm. Whichever zone the player is standing in triggers that arm's corresponding attack event in the animation Blueprint. Each arm has two attack animations — a slam and a swipe — plus a shared clap attack that uses both arms together. Each attack was assigned an integer, and which attack played was chosen by picking a random integer rather than following a fixed pattern. The clap attack was registered in both zones so it could trigger regardless of which side the player was standing on. If the player stays in the same zone, that arm keeps attacking on a cooldown rather than firing every frame.

Mini-boss spawning (attack robot dogs)

The mini-bosses are robot dogs spawned from two points on either side of the BedMonster. Which point spawns a dog is picked with a random integer, with a check in place so the same spawn point can't repeat more than twice in a row.

Spawn timing is tied to the BedMonster's own attacks rather than a flat timer: an attackCount tracks how many times the BedMonster has attacked, and a spawnThreshold (also a random integer) sets how many attacks need to happen before the next dog spawns — once attackCount >= spawnThreshold, a dog spawns and the count resets. This threshold is also gated by the BedMonster's health: if health is at or below 130, the threshold is locked to 1 (a dog spawns after every single attack); above that, it's randomized between 1 and 2. The result is that dog spawns get noticeably more frequent as the BedMonster's health drops, creating a natural difficulty ramp over the course of the fight.

Effectively, the boss "decides" which arm to attack with based on player position rather than tracking and pathing to a target — a workaround built specifically because there was no State Tree or Behavior Tree driving target selection.

Bug

Attacks cutting each other off

While setting up the animation Blueprint, the system would snap straight into the next attack animation before the current one had finished playing, cutting the movement short. The fix was adding a delay matching the exact remaining duration of the in-progress attack animation, so the next attack couldn't trigger until the current one had fully played out.

Technical Challenge: Quest Tracker Sync Bugs

The quest tracker UI (a chalkboard panel that slides in from screen-right when you press Q, showing active and completed quests) was straightforward to build visually — the hard part was keeping it in sync with actual gameplay state. Bugs included quests appearing twice in the list, quests not showing up at all, and completed quests failing to move down into the completed section.

Active and completed quests each live in their own array. A quest is created from one of three trigger conditions: reaching a specific area (detected via a collision box overlap), killing a boss, or collecting a teddy animal. When any condition fires, a quest struct is built with a quest name, description, and an isCompleted boolean, then added to the active quests array. From there, the system calls a function on the quest log widget to add the entry to the chalkboard UI, and finally saves the game so progress persists immediately after each quest update. BP_QuestManager also has a separate function responsible for moving quests over: it checks the active quests array for any entry where isCompleted is true, and if found, removes it from active and adds it to completed.

Bug

The struct-scoping bug

The root cause turned out to be a struct-scoping mistake: when building each quest, I was using a single shared variable of the quest struct type to construct every quest, rather than giving each quest its own independent instance. Because every quest was being written into and read from that same shared variable, the data wasn't properly isolated per quest — which is what produced the duplicate entries, missing entries, and quests that wouldn't move to completed.

The fix was creating a separate quest struct variable for each individual quest, so each one held its own independent data instead of sharing a single backing variable. It's not the most scalable architecture — a more proper fix would be constructing a local struct instance inside the quest-creation function each time and adding copies to the array — but it resolved the sync issues and got the system working correctly.

What I'd Do Differently Now

Knowing what I know now, I'd rebuild the boss AI using Unreal's State Tree (or Behavior Tree) system instead of hand-rolled logic — it would be more maintainable and easier to extend with new attack patterns. I'd also rework the quest struct handling to construct a local instance per quest inside the creation function, instead of relying on separate persistent variables for each one.

At the time I was focused on getting it working correctly under deadline, not on the cleanest implementation.
