Overview

Sleepy Trigger is a First-person action game where a boy falls asleep holding three of his teddy animals and wakes up inside his own dream. Three bosses have stolen the teddies, and the player must fight through the dream world to recover them.

As the gameplay programmer on an eight-person team, I focused on building many of the core gameplay systems that tied the project together. My contributions included the save and persistence system, boss AI, inventory, quest tracking, projectiles, object physics, player movement, animation integration, gameplay UI, and portions of the scene lighting.

Technical Challenge: Full Game State Persistence

This system needed to restore the entire progression state of the game. Player health, collected teddy animals, defeated bosses, destroyed actors, quest progression, stamina, and player settings all had to survive a save/load cycle so the world returned exactly as the player left it.

A SaveData Blueprint stores the game's persistent variables while a BP_QuestManager gathers the current game state, builds the save object, and writes it to disk. Loading follows the same process in reverse, restoring every saved value back into the game.

The more difficult part wasn't saving the variables. It was restoring the world correctly. Progression is built around defeating bosses that unlock the next area. Loading a save had to restore not only whether a boss was defeated, but everything that depended on it: progression obstacles stayed removed, completed encounters remained complete, and the next area stayed accessible. The goal wasn't simply restoring data, it was restoring the player's progress exactly as they had left it.

Technical Challenge: Boss AI Without State Trees

When I built this project, I didn't know about Unreal Engine's State Tree or Behavior Tree systems, so I implemented the boss AI manually.

The BedMonster presented an unusual challenge because it never moves. Instead of chasing the player, both arms remain attached to the bed while the player moves around the arena. Rather than relying on pathfinding, the AI only needed to determine which arm should attack and when.

Zone-based attack selection

The arena is divided into two collision zones, one for each arm. Entering a zone activates the corresponding arm, while each arm randomly selects between its slam and swipe attacks instead of following a fixed sequence. A shared clap attack can trigger from either side, making the fight less predictable. The hands themselves use multiple collision boxes that match the shape of the fingers and palm instead of relying on a single large collision volume, producing more accurate hit detection during each animation.

Difficulty ramp

The encounter also becomes more aggressive as the fight progresses. Robot dogs spawn alongside the boss, with spawn frequency increasing as the BedMonster's health decreases. Rather than spawning on a simple timer, dog spawns are tied to the boss's own attack count, creating a natural difficulty ramp over the course of the fight.

Looking back, I'd rebuild this system today using Unreal's State Tree or Behavior Tree tools. At the time, building it manually taught me a great deal about structuring gameplay logic without relying on Unreal's built-in AI frameworks.

Debugging

Attack animations interrupting each other

While building the animation Blueprint, new attacks could begin before the current animation had finished playing, causing attack animations to snap directly into the next one.

The fix was delaying the next attack until the remaining duration of the current animation had completed. This ensured every attack finished cleanly before another could begin.

Quest tracker synchronization

The quest tracker occasionally displayed duplicate quests, failed to display quests, or didn't correctly move completed objectives into the completed section.

The issue turned out to be a struct-scoping mistake. Every quest was being built from the same shared quest struct instead of its own independent instance, causing multiple quests to overwrite one another's data. Creating a separate quest struct variable for each quest resolved the synchronization issues.

What I'd Do Differently

Knowing what I know now, I'd rebuild the boss AI using Unreal Engine's State Tree or Behavior Tree systems instead of manually managing the logic. The overall behavior would be easier to extend and maintain as the encounter grew more complex.

I'd also redesign the quest system so each quest constructs its own local struct during creation rather than relying on separate persistent variables. While the original implementation worked correctly, the newer approach would scale more cleanly as additional quests were added.
