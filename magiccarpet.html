Trajectory-Driven Procedural Flight System

A flying carpet whose shape emerges from the path it travels, not from frame-by-frame input.

RoleTechnical Artist / Gameplay Systems

EngineUnreal Engine 5.6

ToolsBlueprint, Control Rig, Niagara

  • Control Rig

Unreal Engine 5.6 · Control Rig · Niagara

Overview

This project explores a procedural approach to flight by using trajectory reconstruction instead of traditional skeletal animation.

The goal was to create a flying carpet that naturally communicates momentum, banking, and direction while remaining responsive to player control. Rather than relying on authored animations or directly deforming the skeleton from player input, the carpet's shape is reconstructed from a continuously updated trajectory, allowing its motion to emerge from the path it travels.

The Challenge

The project originally began as a fully procedural Control Rig implementation. Player pitch and roll values were used to directly deform the carpet, creating expressive banking and curvature with very little setup. While the system looked convincing, it exposed an architectural problem as development continued.

The carpet's deformation was reacting to the player's current input while movement followed its own evolving path. During sustained turns and aggressive pitch changes, the visual shape of the carpet could gradually diverge from the trajectory it was actually following.

Instead of adding more procedural logic to compensate, I stepped back and reconsidered the architecture itself.

The Solution

The final implementation is built around a shared trajectory. Player movement continuously generates a path through space, and Control Rig reconstructs the carpet from that trajectory instead of procedurally posing it from instantaneous input. Every major system references the same movement data.

By treating trajectory as the source of truth, the carpet naturally bends, banks, and recovers while remaining visually aligned with its direction of travel.

Technical Highlights

Trajectory Reconstruction

Instead of directly rotating bones from player input, the carpet is reconstructed from sampled points along its trajectory. This approach allows it to:

  • Maintain curvature through sustained turns
  • Naturally propagate banking along its length
  • Keep deformation aligned with movement
  • Produce stable results across different traversal speeds

Layered Camera System

The camera is separated into three independent responsibilities:

  • Camera Target — represents the desired direction of travel.
  • Camera Anchor — smoothly commits toward that direction.
  • Camera — applies additional positional and rotational lag to create inertia and reduce high-frequency motion.

Separating these layers allows the camera feel to evolve independently from the flight system.

Procedural Secondary Motion

Secondary flutter is generated procedurally and evaluated relative to the carpet's trajectory instead of a fixed world orientation. This keeps idle motion consistent while preserving readability during active flight.

Niagara Integration

Ribbon trails attached to the edges of the carpet reinforce:

  • Banking
  • Curvature
  • Speed
  • Direction changes

Rather than acting as exhaust effects, the ribbons visualize the path driving the system.

Development Process

The biggest lesson from this project was realizing that procedural animation alone is not always the best representation of movement.

My first implementation focused on generating deformation directly from player input. While visually expressive, it created multiple interpretations of direction that gradually drifted apart during complex motion.

Rebuilding the system around a shared trajectory simplified the architecture and naturally synchronized deformation, camera behavior, and visual effects without increasing procedural complexity.

What I Learned

This project changed the way I approach procedural animation.

Instead of asking, "How should this object animate?" I started asking, "What information should every system share?"

By allowing deformation, camera behavior, and visual effects to reference the same trajectory, each system remains independent while contributing to a single, cohesive representation of movement.

Summary

Trajectory-Driven Procedural Flight System is an exploration of procedural motion built around a simple idea: movement should be described once and interpreted by every downstream system. By reconstructing the carpet from a shared trajectory instead of directly deforming it from player input, the project produces motion that is expressive, cohesive, and easier to reason about while demonstrating an iterative approach to technical problem solving.
